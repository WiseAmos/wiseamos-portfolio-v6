02 · How I work

A few principles
that keep the bug count down.

Most of what I build lives close to the metal — autonomous vehicles, trading systems, face-locked laptops. The tolerance for "it kinda works" is zero. Here's the loop.

01
Step one

Get the constraints wrong first.

Most engineering disasters start with a clean problem statement. The work is in finding the messy one. I read the spec, then I try to break it. Latency budget, failure modes, the boring middle of the day when nothing is on fire. By the time I write code I've usually decided which abstractions to throw away.

02
Step two

Build the smallest thing that proves the model.

I don't write a thousand lines and hope. The first commit is usually a single end-to-end path with the dumbest possible implementation. Then I instrument it, run it against real data, and watch where it falls over. Once I know what's actually slow or wrong, I rewrite the parts that matter.

03
Step three

Optimise for the human in the loop.

Most software is operated by someone at 2am. I write logs a stranger can read, dashboards a tired person can scan, error messages that name the actual problem. If a system is hard to operate, it's not finished — it's just demo-ready.

04
Step four

Ship, then watch it fail in production.

The point of building is running. Once it's live I watch the metrics, read the errors, talk to the users. The first week of production teaches more than a month of staging ever did. Then I do the work over again, but with real numbers.

03 · About

Engineer based in Singapore.

I'm Amos — I build systems where correctness isn't optional. High-frequency trading, autonomous vehicles, computer vision products. The kind of work where a missed frame costs money or a wrong label embarrasses the user.

My day-to-day is Python, TypeScript, and a bit of Rust when the latency budget tightens. I work in tight loops — spec, prototype, ship, measure, repeat. I prefer the boring, debuggable solution to the elegant one. Most of my best work is invisible — it's a thing that doesn't break.

I read more than I code, write a little, and run long enough to think. Available for contract work and the occasional research collaboration.

PRINCIPLE / 01

Correctness over cleverness

The clever solution that's wrong twice is worse than the boring one that's right.

PRINCIPLE / 02

Real data beats test fixtures

If I can't run it on production traffic, I don't know if it works.

PRINCIPLE / 03

Operability is a feature

Logs, dashboards, error messages — not a polish step, the work itself.

PRINCIPLE / 04

Ship, then iterate

Production is the only classroom that matters. Get there fast.
